# 開発プロセス

## 概要
アジャイル、ウォーターフォール、イテレーション手法を適切に選択・カスタマイズし、高品質でスケーラブルな開発体制を実現します。

## アジャイル開発（変化への対応重視）

**本質**：定期的に動作するプロダクトをデリバリ → ユーザーフィードバック基に方向修正 → 繰り返し

**サイクル**：MVP定義 → 短い反復（1～2週間） → デモ・フィードバック収集 → 学習・改善・次反復へ

MVP（最小限の機能セット）から開始し、ユーザー反応を直接観察。想定との乖離発見、改善方向を具体化。

```python
class AgileProcess:
    def define_mvp(self):
        return {
            'core_features': ['ユーザー登録・ログイン', '基本検索', 'お気に入り'],
            'excluded': ['高度な分析', '複雑な推薦', 'マルチ言語'],
            'rationale': '最速でユーザーに価値提供、フィードバック獲得'
        }

    def analyze_feedback(self, user_feedback_data):
        return {
            'quantitative': {'usage_rate': '機能A: 85%', 'nps': 65},
            'qualitative': {'interviews': 'ユーザーが価値を感じる機能特定'},
            'next_actions': ['低利用率機能改善', 'NPS向上施策継続']
        }
```

## ウォーターフォール開発（安定性重視）

**適用シーン**：要件明確・変更少ない、規制厳格、大規模チーム・複雑な依存関係

**フロー**：要件定義 → 基本設計（アーキテクチャ・DB） → 詳細設計 → 実装 → テスト → 本番化

前の工程が終わらないと次が始まらない。後期問題発覚時の修正コストが膨大なため、早期リスク検出が重要。

クリティカルパス（DB設計、セキュリティアーキテクチャ、テストシナリオ）を特定し、ゲートレビューで各フェーズ完成度を検証。

```python
class WaterfallProcess:
    def validate_phase_completion(self, phase_name):
        if phase_name == 'requirements':
            return ['機能要件', '非機能要件', 'ステークホルダー承認', 'トレーサビリティ']
        elif phase_name == 'design':
            return ['アーキテクチャ図', 'DB設計', 'API仕様', 'セキュリティ設計']

    def identify_and_mitigate_risks(self):
        return [
            {'risk': '追加要望', 'mitigation': 'チェンジマネジメント厳格化'},
            {'risk': '実装不可能性発見', 'mitigation': 'プロトタイプ検証'},
            {'risk': '後期欠陥発見', 'mitigation': 'レビュー・単体テスト厳格化'}
        ]
```

## イテレーション（反復）の価値

PDCA（Plan → Do → Check → Act）サイクル：目標設定 → 実行 → 測定 → 定着化

複数回反復で見積もり精度向上（1反復:±50% → 3反復:±30% → 7反復:±15%）、プロセス成熟度が段階的に上昇。

ユーザーフィードバック、チーム振り返り、市場シグナルから継続的に改善。

## プロセス選択ガイド

| 要件 | アジャイル | ウォーターフォール |
|------|----------|------------------|
| 要件明確性 | 不明確・変化有 | 明確・固定 |
| 顧客関与度 | 高い | 低い |
| 納期圧力 | 柔軟 | 厳格 |
| 規制要件 | 緩い | 厳格 |
| チーム規模 | 小～中 | 大規模 |

**ハイブリッド**：全体設計フェーズはウォーターフォール、実装以降はアジャイル（スケーアジャイル）を採用することが多い。
